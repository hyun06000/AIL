# Round 2 Benchmark Analysis — 2026-04-21

## Summary

Round 2 added a **FORBIDDEN SYNTAX** block to both base and tutorial authoring prompts,
blocking common parse-failure patterns identified from Round 1: `[Number]`/`[Text]` list
type annotations, dict literals `{}`, keyword arguments `fn(x=5)`, `**` exponent operator,
and `import stdlib.X` dot notation.

**Result: R2/C4 (fine-tuned + no few-shot + FORBIDDEN) = 64% answer rate — first time AIL
beats the Python baseline (56%).**

## Key Findings

### 1. FORBIDDEN SYNTAX is highly effective — for fine-tuned model without few-shot

R1/C4 → R2/C4: parse 58% → **72%** (+14pp), answer 48% → **64%** (+16pp).

The improvement is concentrated in Cat A (computation) and Cat C (hybrid), exactly the
categories where syntax errors were most frequent in Round 1.

### 2. FORBIDDEN SYNTAX + tutorial = regression for fine-tuned model

R1/C5 → R2/C5: answer rate **52% → 44%** (−8pp). Regressions:

| Case | Root cause |
|------|-----------|
| A06  | Tutorial pushed model toward `pure fn`; `sum_list` not in purity registry → PurityError |
| A10  | `reverse(split(s))` returns list; compared to string → wrong |
| B04  | LLM judgment variance (borderline formal/informal case) |
| B11  | Model returned `"fr"` (ISO code) instead of `"French"` — format shift from longer prompt |

The tutorial prompt combined with FORBIDDEN SYNTAX overloads the fine-tuned model,
causing pattern confusion. The FORBIDDEN block alone (C4 style) works better.

### 3. FORBIDDEN SYNTAX has no effect on base model

R1/C2 vs R2/C2: identical (48% / 60% parse). Base model ignores the forbidden block,
still generating the same invalid patterns. Fine-tuning is essential for this to matter.

### 4. Cat C (hybrid) improvement is most significant

R1/C4 Cat C: 45% → R2/C4 Cat C: **60%** (+15pp). Hybrid tasks require both `fn` and
`intent` in one program — the parse failures from wrong syntax were disproportionately
blocking these.

## Optimal Configuration

**Fine-tuned model (ail-coder:7b-v3) + no few-shot + FORBIDDEN SYNTAX = 64%**

This is the configuration to use for:
- All future benchmarks as the AIL baseline
- `ail ask` production default (when running with fine-tuned model)
- Fine-tuning data generation for v4

Do NOT combine tutorial few-shot with the fine-tuned model — it regresses performance.

## Python Baseline Note

Python answer rate in R2 (40%) is lower than R1 (56%). This is because R2 C4/C5 used
`ail-coder:7b-v3` for Python generation too (same model, different prompt), while R1 C1
used `qwen2.5-coder:7b-base`. The fine-tuned model is worse at Python — it has been
specialized toward AIL. The fair comparison is:

- AIL fine-tuned: **64%**
- Python with same model: 40%
- Python with base model (honest baseline): **56%**

AIL beats both.

## Next Steps

1. **Cat A recovery**: A06 (`pure fn` + `sum_list`) reveals that stdlib builtins need
   to be registered in the purity checker. Fix: add stdlib functions to the pure-trusted
   list, or adjust FORBIDDEN to discourage `pure fn` for stdlib-heavy computations.

2. **Cat C gap**: Still 60% vs Python's 80% (base model). Primary cause is fn+intent
   interleaving — models generate fn-only or intent-only when both are needed. Consider
   adding a hybrid few-shot example to the base (non-tutorial) prompt.

3. **v4 fine-tuning**: Now have clear failure patterns to add to training data:
   - `sum_list` in pure fn context (A06 pattern)
   - Palindrome via `split`+`reverse` (A10 pattern)
   - Language name vs code (B11 pattern)

4. **Target**: R3/C4 ≥ 70% answer rate.
