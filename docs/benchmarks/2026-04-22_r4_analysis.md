# R4 Benchmark Analysis — ail-coder:7b-v4 regression

**Date:** 2026-04-22
**Model:** ail-coder:7b-v4 (qwen2.5-coder-7b-instruct + QLoRA r=16, 260 samples, 3 epochs)
**Backend:** vLLM on homeblack (GGUF via Ollama blob path)
**Condition:** C4 (ft/nofs) — same model for AIL and Python authoring
**Result file:** `2026-04-22_r4_cond4_finetuned_nofewshot.json`

## Verdict

**v4 is a regression. Keep v3 as the serving model.**

- Overall AIL answer rate: **68%** (v3 was 70%, Δ = −2pp)
- Cat A improved +20pp; Cat B regressed −27pp; Cat C unchanged
- **Two silent LLM skips emerged** in Cat B — breaks AIL's structural-safety claim

## The critical finding — silent LLM skips

v3 used LLM on 100% of Cat B tasks (15/15). v4 dropped to 87% (13/15). **Two pure-judgment tasks were solved without calling the LLM.**

This directly contradicts AIL's core claim that "`intent` is a dispatch declaration — the runtime routes every declared intent through the model adapter, and there is no AIL syntax for declare the intent and then don't call it."

The grammar claim is still true. But v4's fine-tune now produces source code that doesn't *declare* intent where intent is needed — it writes pure-fn pipelines for judgment tasks. The symptom: answer rate drops, uses_llm drops, grammar guarantee is bypassed at the authoring stage.

### Specific Cat B regressions (v3 ✅ → v4 ❌)

| ID | Task | v3 | v4 | v4 failure mode |
|---|---|---|---|---|
| B03 | Summarize sentence | ✅ intent+LLM | ❌ no LLM | silent skip |
| B08 | Determine topic | ✅ intent+LLM | ❌ no LLM | silent skip |
| B11 | Classify language | ✅ intent+LLM | ❌ LLM called, wrong output | returned "Bonjour, je vais bien..." instead of "French" |
| B14 | Fact or opinion | ✅ intent+LLM | ❌ LLM called, wrong output | returned literal "fact_or_opinion" |

B03 and B08 are the silent skips. B11 and B14 did call the LLM but returned wrong-format answers — likely an artifact of the tighter FORBIDDEN SYNTAX prompt constraining output shape.

### Cat A improvements (v3 ❌ → v4 ✅)

| ID | Task |
|---|---|
| A04 | Find largest number |
| A10 | Palindrome check |
| A12 | Dedupe list |

These are exactly the kind of compound-arithmetic problems that `08_compound_ko.jsonl` (16 Korean samples, all pure_fn) targets.

## Root cause hypothesis

The v4 training set pulled the distribution toward fn-heavy solutions:

- `08_compound_ko.jsonl`: 16 Korean compound-arithmetic samples — **all pure_fn**
- `09_r3_fixes.jsonl`: 11 samples targeting R3 failures — **rejected by validator** (`source_of_sample: "r3_fixes"` not in allowlist)

Net effect: +16 pure_fn, −11 pure_intent/hybrid that were supposed to reinforce intent use. The distribution shifted toward "try to solve with fn first."

Validation log confirms:
```
[FAIL] r3fix_000 ... bad source_of_sample: 'r3_fixes'
[FAIL] r3fix_001 ... bad source_of_sample: 'r3_fixes'
... (11 total rejections)
TOTAL  260/273 passed
```

## What v4 taught us

1. **Category balance matters in fine-tune data.** A 16-sample skew toward pure_fn was enough to cause a 27pp regression on Cat B.
2. **Korean compound samples help Cat A.** The +20pp is real and reproducible. The problem is that they shipped without a pure_intent counterweight.
3. **The validator allowlist is a trap.** Intentional R3 fixes were silently dropped. The validator should list all legitimate `source_of_sample` tags or fail loudly when a new one appears.

## Decision

- **Serving model: v3 (unchanged).** v4 is archived as an experiment.
- **Do not promote v4 in `ail ask` or docs.** Current user-facing docs reference `ail-coder:7b-v3`.
- **v5 plan (not scheduled):**
  - Fix validator allowlist to accept `r3_fixes` or migrate those 11 samples to a legitimate tag
  - Offset the 16 Korean pure_fn samples with equal pure_intent/hybrid samples (target 16+ of each)
  - Retrain with 280+ balanced samples
  - R5 benchmark to confirm
