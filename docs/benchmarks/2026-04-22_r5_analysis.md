# R5 Benchmark Analysis — ail-coder:7b-v5 (single-line training experiment)

**Date:** 2026-04-22
**Model:** ail-coder:7b-v5 (qwen2.5-coder-7b-instruct + QLoRA r=16, 291 samples, 3 epochs)
**Training format:** single-line AIL (no indentation, no newlines, comments stripped)
**Backend:** vLLM on homeblack, Q4_K_M GGUF via peft-merge path
**Condition:** C4 (ft/nofs) — same model for AIL and Python authoring
**Result file:** `2026-04-22_r5_cond4_finetuned_nofewshot.json`

## Verdict

**v5 is a severe regression. Hypothesis REJECTED: single-line training does not preserve generation quality.**

- Overall AIL answer rate: **42%** (v3: 70%, v4: 68%). −28pp from v3.
- Cat C (hybrid) crashed from 70% → **20%**. A 50pp drop.
- Keep v3 as the serving model. v5 is archived as a failed experiment.

## Headline numbers

| Metric | v5 (R5) | v4 (R4) | v3 (R3) | Δ vs R3 |
|---|---|---|---|---|
| AIL parse success | **50%** | 80% | 80% | **−30pp** |
| AIL answer correctness | **42%** | 68% | 70% | **−28pp** |
| Python answer correctness | 2% | 32% | 40% | −38pp |
| AIL avg retries | **1.68** | 0.54 | — | **3× worse** |
| AIL avg completion tokens | 59 | 89 | — | −34% |
| AIL error handling miss | 0% | 0% | 0% | = (grammar) |

Retries tripled and completions got shorter — the model is failing to parse, getting feedback, and failing again. The feedback loop could not recover.

## Per-category damage

| Category | N | v3 ans | v5 ans | Δ | v5 uses_llm |
|---|---|---|---|---|---|
| A — pure computation | 15 | 60% | 60% | 0pp | 0% |
| B — pure judgment | 15 | 80% | 53% | −27pp | 73% |
| C — hybrid | 20 | 70% | **20%** | **−50pp** | 15% |

Cat A held up. Cat B and especially Cat C collapsed. This correlates with program complexity: simple one-function programs survive single-line training; multi-function programs (which Cat C requires) break hardest.

## Root cause — the leading-quote artifact

The primary failure mode: the model outputs AIL code **wrapped in a leading double quote**, causing the parser to reject it as a top-level string literal.

Representative failures:

```
A08: "pure fn is_long(c: Text) -> Boolean { return length(c) > 3 } ...
     ↑ leading " breaks parse

A10: "pure fn reverse_text(s: Text) -> Text { ...
B01: "intent classify_sentiment(text: Text) -> Text { goal: ...
A15: "pure fn is_divisible_by_four(n: Number) -> Boolean { ...
```

The parse error in each case: `ParseError: unexpected top-level token STRING('...')`.

### Why this happens

Qwen's pretraining distribution contains massive amounts of Python, where single-line code in training/inference contexts is typically a **string literal** inside another program (`code = "def foo(): pass"`). When we fine-tuned on single-line AIL, the model's existing heuristic — "this one-line code-looking thing is probably a quoted string" — activated during generation. The fine-tune weights were strong enough to produce AIL keywords but weak enough that the surface-level "this should be quoted" habit bled through on harder programs.

Cat C's 50pp collapse is especially damning: complex multi-function programs look the most like "big strings of code" to the model, so the quoting artifact triggers more often.

Secondary failure mode: on A09 (FizzBuzz), the model produced **empty output** (`src: ''`). Likely truncation or early-stop triggered by an out-of-distribution prompt under the new training format.

## Why this is a useful finding (not just a failure)

The v5 experiment was designed to test the hypothesis: "AIL's grammar is brace-delimited, so indentation is purely cosmetic. Training on single-line code should preserve quality while reducing tokens." The hypothesis was **clean, testable, and wrong** — and we now have concrete data for why.

Three takeaways:

1. **AIL's parser accepts single-line code.** That is still true. The problem is not the language; it is the training distribution shift.
2. **The model's pretraining prior dominates even after fine-tune.** Qwen has seen millions of `"one-line python"` strings; our 291 single-line AIL samples can't fully displace that prior.
3. **A format change during training isn't free.** The 9% token efficiency on training tokens cost us 28pp in answer correctness. That trade is never worth it.

## v6 recommendation — if we retry at all

If a v6 experiment is worth running, the right variable is **strip-indent, not single-line**:

- `strip-indent` removes only leading whitespace per line; newlines are preserved.
- Token savings are roughly the same as single-line (measured ~9% on Qwen tokenizer for both modes).
- Newlines act as statement separators the model already uses when it generates multi-function code.
- Output looks like AIL code with sloppy formatting, not like a quoted string literal.

Test protocol for v6:
1. Regenerate chatml with `--flatten=strip-indent`.
2. Same dataset (291 samples, validator allowlist already fixed).
3. Same hyperparams (LoRA r=16, 3 epochs).
4. Benchmark with R3 as the must-beat bar.

If v6 also regresses, the full conclusion is: **don't reformat training data at all**. Keep the original indented form and invest training-data effort in content quality instead of formatting economy.

## Decision

- **Serving model: v3.** Unchanged.
- **Do not promote v5 in any user-facing doc.** The public readme continues to reference v3.
- **Archive v5 as an experiment.** Ollama registration kept for reproducibility (`ail-coder:7b-v5`, ID `2295279d075a`).
- **v6 is conditional** — only run it if the `strip-indent` hypothesis is worth testing. Not urgent.

## What does this say about Q16 (comments in AIL)?

The v5 experiment also stripped `//` line comments from training data. Because v5 regressed so badly for *other* reasons (single-line), we cannot isolate the comment-stripping effect. Q16 remains open. Any future strip-indent experiment should keep comments or strip them as an explicit separate variable.
