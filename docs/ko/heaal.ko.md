# HEAAL: Harness Engineering As A Language

*AI 코드 생성 안전성을 위한 새로운 패러다임*

---

## 한 문장으로 요약

2026년, AI 에이전트를 만드는 모든 팀이 Python 위에 안전장치(하네스)를 직접 만들고 있습니다 — 린터, AGENTS.md, pre-commit hook, CI 규칙. **HEAAL은 묻습니다: 하네스 자체가 언어라면?**

---

## 배경: Agent = Model + Harness

Mitchell Hashimoto가 2026년 초에 정립한 공식입니다. 모델은 두뇌이고, 하네스는 그 외의 모든 것 — 제약, 피드백 루프, 검증, 문서입니다. Martin Fowler는 하네스를 두 종류로 나눴습니다:

```
  가이드 (행동 전 제어)                센서 (행동 후 검증)
  ────────────────────               ──────────────────
  시스템 프롬프트                      린터
  AGENTS.md                           테스트 스위트
  아키텍처 제약                        출력 검증기
  코딩 컨벤션                          LLM-as-judge
```

2026년의 핵심 발견: **모델을 바꾸는 것보다 하네스를 바꾸는 게 더 효과적입니다.** LangChain은 모델 변경 없이 하네스만 바꿔서 Terminal Bench 2.0에서 30위 → 5위로 뛰었습니다. Stanford의 Meta-Harness 연구는 하네스 최적화만으로 작은 모델이 큰 모델을 이길 수 있음을 보였습니다.

## AI 코드 안전성의 세 단계

```
  1단계: 바이브 코딩 (2025)
  ┌──────────────────────────────────────┐
  │  사람이 말함  →  AI가 코드 작성  →  바로 출시   │
  │  안전성: 없음                                │
  └──────────────────────────────────────┘

  2단계: 하네스 엔지니어링 (2026)
  ┌──────────────────────────────────────┐
  │  사람이 말함  →  AI가 작성  →  하네스가 검사     │
  │  안전성: 하네스 품질에 의존                     │
  │  비용: 구축에 수일, 지속적 유지보수 필요         │
  └──────────────────────────────────────┘

  3단계: HEAAL
  ┌──────────────────────────────────────┐
  │  사람이 말함  →  AI가 작성  →  언어 자체가 하네스 │
  │  안전성: 구조적 — 파서가 강제                   │
  │  비용: 설정 없음, 유지보수 없음                  │
  └──────────────────────────────────────┘
```

비유: **C는 원시 포인터를 줍니다. C++는 스마트 포인터를 관례로 제공합니다. Rust의 빌림 검사기는 메모리 안전을 컴파일러가 보장합니다.** HEAAL은 AI 코드 생성에서 Rust가 메모리 관리에서 한 것과 같은 일을 합니다 — 안전 보장을 관례에서 문법으로 옮깁니다.

## HEAAL이 실제로 의미하는 것

HEAAL 언어는 가이드와 센서를 문법 자체에 내장합니다:

```
  가이드 (문법적 제약)                  센서 (내장된 관측성)
  ─────────────────────               ──────────────────────
  ✗ while 없음 → 무한 루프 불가능       모든 값에 확신도(confidence)
  ✗ pure fn → 부작용 컴파일 타임 거부   출처 추적 (origin_of)
  ✗ Result 타입 → 에러가 값임          결정 추적 (전체 감사 로그)
  ✗ evolve는 rollback_on 필수         적응적 메트릭 모니터링
  ✗ rewrite → 사람 검토 강제           관측치 기반 보정(calibration)
```

설정할 것 없음. 유지보수할 것 없음. 낡아질 것 없음.

## 기존 방식 vs HEAAL: 비교

```
  기존 방식 (Python + 외부 하네스)

    모델
      ↓
    AGENTS.md ←── 직접 작성하고 유지보수
    커스텀 린터 ←── 프로젝트마다 설정
    Pre-commit hooks ←── 레포마다 세팅
    CI/CD 규칙 ←── 팀마다 구축
    LLM-as-judge ←── 별도 배포
      ↓
    Python 코드 (내장된 안전장치 없음)


  HEAAL (언어가 하네스)

    모델
      ↓
    HEAAL 언어 ←── 안전이 문법에 내장
      ↓
    안전성 + 관측성이 내장된 코드
```

| | 기존 방식 | HEAAL |
|---|---|---|
| 설정 비용 | 프로젝트당 수일 | 없음 |
| 유지보수 | 지속적 (규칙 낡음, 문서 부패) | 없음 — 문법은 낡지 않음 |
| 무한 루프 | 타임아웃으로 잡음 | 불가능 — `while` 없음 |
| 순수 코드의 부작용 | 린터로 잡음 | 불가능 — `pure fn` 컴파일 타임 |
| 에러 처리 | 관례 기반 | 강제 — `Result` 타입 |
| 관측성 | 커스텀 계측 | 내장 — 확신도 + 출처 + 추적 |
| 모델 종속 | 모델별 하네스 조정 | 어떤 모델이든 — 언어가 하네스 |
| 이식성 | 팀마다 독자적 | 스펙으로 표준화 |

## AIL: 최초의 HEAAL 언어

AIL(AI-Intent Language)은 HEAAL 패러다임의 레퍼런스 구현입니다. 두 가지 핵심 구성 요소가 철학을 구현합니다:

**`fn`** — 순수 결정적 계산. LLM 없음, 부작용 없음, 확신도 1.0.
**`intent`** — 판단이 필요할 때 언어 모델에 위임.

AI 저자가 둘 중 하나를 선택합니다. 언어가 그 선택의 결과를 강제합니다.

```
fn count_words(text: Text) -> Number {      ← 계산: fn
    return length(split(text, " "))
}

intent classify(text: Text) -> Text {       ← 판단: intent
    goal: sentiment label
}

entry main(text: Text) {
    count = count_words(text)               ← LLM 없음, 즉시, 무료
    mood = classify(text, "pos_neg")        ← LLM, 1-3초, 토큰 비용
    return join([mood, to_text(count)], " ")
}
```

**실증적 발견:** Claude Sonnet이 AIL 관련 학습이나 파인튜닝 없이도, 동일한 태스크에서 Python과 동등하거나 그 이상의 안전성으로 AIL 코드를 생성했습니다. 모델이 똑똑해진 게 아닙니다. 하네스가 오류의 범주를 제거한 것입니다.

---

## HEAAL이 주장하지 않는 것

- **Python보다 빠르다고 주장하지 않습니다.** 병목은 실행 속도가 아니라 LLM 지연 시간입니다.
- **Python을 대체한다고 주장하지 않습니다.** Python의 생태계(numpy, pandas, Django)는 대체 불가능합니다.
- **하네스가 불필요하다고 주장하지 않습니다.** HEAAL은 하네스*입니다* — 설정이 아닌 문법으로 된 하네스.
- **파인튜닝이 필요 없다고 주장하지 않습니다.** 다만, 내장된 하네스가 이미 충분한지 먼저 확인하라는 것입니다.

---

## 에이전트 아키텍처에 주는 시사점

모델이 상품화되면(상위 모델 간 벤치마크 격차가 줄어들면), **하네스가 경쟁 우위**가 됩니다.

AGENTS.md와 린터로 만든 하네스는 독자적이고, 깨지기 쉽고, 이식 불가능합니다. 언어*인* 하네스는 표준화되고, 이식 가능하고, 유지보수가 필요 없습니다. 모델을 자유롭게 교체하세요. 안전성은 그대로입니다.

멀티 에이전트 시스템에서: 각 에이전트가 HEAAL 언어로 작성합니다. 각 에이전트의 출력은 구조적으로 안전합니다. 오케스트레이터는 안전성을 검증할 필요 없습니다 — 문법이 이미 했으니까요.
