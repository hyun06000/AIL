# AIL FAQ — 실무자가 궁금한 것만

🇬🇧 English: [`../why-ail-faq.md`](../why-ail-faq.md)

이 문서는 AIL 도입을 검토하는 개발자/팀이 실제로 던지는 질문에,
2026년 4월 벤치마크 실측 데이터로 답합니다. 수사 없음, 벤치마크
JSON 스냅샷은 [`../benchmarks/`](../benchmarks/)에 전부 커밋되어
있습니다.

---

## 한 줄 요약

> 같은 7B 모델을 썼을 때, **AIL로 작성된 코드는 정답 비율 70%, Python은 48%**.
> **AIL은 에러 핸들링을 문법이 강제해서 누락률 0%**. Python은 44%가 누락.

이 두 문장이 전부입니다. 나머지는 "어떤 조건에서, 얼마나" 입니다.

---

## Q1. 토큰은 얼마나 아낄 수 있나?

**측정값:** `ail-coder:7b-v3` 으로 50 프롬프트 벤치마크를 돌린 결과 **AIL 은 총 37 번 LLM 을 호출** 했습니다. 전부 프롬프트가 실제로 모델 판단을 요구한 자리에만 — 순수 계산 과제에는 0 건.

**카테고리별 평균:**

| 카테고리 | 작업 성격 | AIL 호출/task (실측) |
|---|---|---|
| A (15 프롬프트) | 순수 계산 | **0.07** (≈0 — 라우팅 오류 1 건) |
| B (15 프롬프트) | 순수 판단 | 0.93 (15 건 중 14 건 올바르게 LLM 호출) |
| C (20 프롬프트) | 하이브리드 (계산 + 판단) | 1.10 |

**0.07 이 왜 중요한가:** 15 개 순수 계산 프롬프트에서 AIL 은 LLM 을 사실상 안 부릅니다. 모든 서브태스크를 LLM 으로 돌리는 naive 에이전트 셋업 ("LangChain 기본") 은 같은 프롬프트들에 대해 적어도 태스크당 1 번씩 LLM 을 부릅니다 — 즉, 순수 계산 태스크가 많을수록 토큰 차이는 (태스크 수 × 호출당 비용) 으로 선형 증가.

**정확한 절감량은 비교 대상에 따라 다릅니다:**

AIL 은 fn/intent 분할을 명시적으로 하기 때문에, 호출 수는 과제 모양을 따라갑니다 — 저자 규율에 의존하지 않습니다. 그래서 "얼마나 아끼는가" 는 **무엇을 대체하느냐** 에 달렸습니다:

- **AIL 자기 자신 대비:** 비교 대상 없음.
- **신중하게 LLM 라우팅을 수동으로 하는 Python 파이프라인 대비:** 절감 없음 — 잘 짠 Python 저자는 AIL 과 같은 호출 예산을 씁니다.
- **이 벤치마크에서 같은 7B 가 작성한 Python 대비:** AIL 이 Python 보다 호출을 **더** 씁니다 (37 vs 18). Python 이 호출을 덜 하는 이유는 필요한 LLM 을 조용히 스킵하기 때문 — 그 대가로 정답률이 26% 떨어집니다 (Q2 참고).
- **모든 서브태스크를 LLM 으로 보내는 naive 에이전트 대비:** 1 번의 LLM 판단이 필요한 하이브리드 과제에서 그런 셋업은 ≥ 2 번 (계획 + 실행) 을 쓰고 흔히 더 씁니다. AIL 이 눈에 띄게 덜 씁니다. 정확한 비율은 에이전트 구현과 과제 모양에 의존합니다 — 벤치마크는 이 셋업을 직접 측정하지 않으므로, 특정 "N× 절감" 수치는 측정이 아니라 **추정**.

**정리:** 잘 조율된 Python 파이프라인 (사람이 신중히 라우팅) 과 AIL 중 고를 때 토큰 절감은 논거가 아닙니다 — Q2 (정답성) 와 Q3 (안전성) 가 논거입니다. 모든 서브태스크에 LLM 을 거는 에이전트 프레임워크 vs AIL 이면 AIL 이 의미 있게 덜 씁니다 — 라우팅이 모델의 런타임 선택이 아니라 **구조적** 이기 때문.

코드 작성 비용 (fine-tuned 7B 를 3070 로컬에서 돌리는) 은 포함하지 않았습니다 — 추론 시점에 실질적으로 0.

## Q2. 품질은 진짜 더 좋은가?

**예 — 같은 모델로 전체 정답 기준 22pp 차이.**

| 지표 | AIL | Python |
|---|---|---|
| 최종 정답 비율 (50 task) | **70%** | 48% |
| `pure_intent` 카테고리 정답 (15 task) | **80%** | 13% |
| 파싱 성공 | **78%** | 54% |

세 숫자 전부 `ail-coder:7b-v3` 으로 50 개 같은 프롬프트를 돌린 한 런에서 나옵니다 — 한쪽은 AIL 로, 다른 쪽은 Python 으로 작성.

**이 격차는 대부분 Python 의 silent LLM-skip 에서 옵니다.** ground truth 가 LLM 호출을 요구한 35 프롬프트 (B + C) 에서, 파싱에 성공한 Python 프로그램은 두 갈래로 갈렸습니다:

- 35 개 중 18 개가 파싱됐습니다. 그 중 12 개 (B:3 + C:9) 는 소스에 **LLM 호출 시도가 아예 없었습니다** — 저자는 문제를 문자열 연산/키워드 매칭으로 풀 수 있다고 선언하고, 런타임은 그걸 실행해 틀린 답을 냈습니다. 이 벤치마크의 유명한 사례는 "수동태로 바꾸기" 프롬프트에서 Python 이 `{object} was {verb} by {subject}` 를 하드코딩해 "chased was cat by The" 를 출력한 것.
- 나머지 6 개 (18 - 12) 는 실제로 LLM 호출을 시도했고 대체로 올바른 답을 얻었습니다.

AIL 은 silent skip 을 할 수 없습니다: `intent classify_sentiment(...)` 는 디스패치 선언이고, 런타임이 모델 어댑터로 라우팅합니다. 저자에겐 "intent 를 선언하고 호출을 스킵" 할 문법이 없습니다.

## Q3. Python보다 안전하다는 건 구체적으로 뭐가?

측정된 3가지 구조적 차이:

| 위험 | Python | AIL | 이유 |
|---|---|---|---|
| 에러 핸들링 누락 (`to_number` 실패 등) | 44% | **0%** | AIL의 `Result` 타입이 `is_ok()` / `unwrap()` 을 강제 |
| 무한 루프 | 런타임 발견 | **불가능** | AIL에 `while`이 없음 |
| pure 함수 안의 몰래 I/O | mypy로 못 잡음 | **파싱 거부** | `pure fn` 안에서 `intent` / `perform` 호출 불가 |

**44% → 0% 격차가 왜 중요한가, 그리고 왜 더 강한 모델로 해결이 안 되는가:**

AI는 코드를 확률로 생성합니다. 학습 데이터에서 `int(x)`, `json.loads(s)`, `open(f)` 같은 함수가 에러 핸들링 없이 쓰인 예제가 압도적으로 많기 때문에, 모델은 실패 가능한 연산을 자연스럽게 처리 없이 내놓습니다. 사람은 경험으로 "이 함수가 터질 수 있다"는 걸 압니다. AI는 그걸 확률로만 알고, 종종 빠뜨립니다.

"프론티어 모델 쓰면 되지 않냐?"의 반례: Sonnet 4.6은 테스트한 모델 중 가장 강하고 LLM 라우팅을 100% 정확히 합니다. 그럼에도 Python 코드에서 실패 가능한 연산의 70%를 에러 핸들링 없이 씁니다. 모델이 강해진다고 이 수치가 수렴하지 않습니다. Python이 `int(x)`를 에러 핸들링 없이 쓸 수 있도록 허용하는 한, 이 문제는 남습니다.

AI가 코드를 생성하고 자율적으로 실행하는 파이프라인에서 이 누락은 더 위험합니다. 사람이 리뷰하면 `try/except`가 없는 걸 발견합니다. 하지만 AI가 생성→실행→결과 반환을 자동으로 처리하면, 잘못된 값이 파이프라인 다음 단계로 조용히 넘어갑니다. 아무도 모릅니다.

AIL에서는 `to_number(x)`가 `Result`를 반환하고, `is_ok()` 호출 없이 `unwrap()`을 쓰면 파서가 거부합니다. 에러 핸들링은 모델이 기억해야 할 것이 아니라 문법이 강제하는 것입니다.

## Q4. 실행 속도는?

AIL이 **느립니다**. task당 평균:

- AIL: 4.8초
- Python: 1.9초

이유 2개:
1. AIL이 LLM을 더 "정직하게" 부릅니다 — Python은 안 불러야 할 때만
   빠릅니다.
2. AIL 런타임은 trace, provenance, confidence 추적에 오버헤드가
   있습니다. 현재 구현은 Python reference 구현이고 Go 구현은 Phase-0
   subset.

**어떤 워크로드에 크리티컬한가:**
- 저지연 챗봇 응답 → 별로 안 맞음
- 배치 처리, 데이터 파이프라인, 코드 에이전트 → 거의 영향 없음

## Q5. 어떤 조건에서 AIL을 선택해야 하나?

**선택:**
- ✅ AI 가 코드를 작성하고 실행하는 파이프라인이 있을 때
- ✅ 계산 + 판단이 섞인 과제가 많을 때 (하이브리드 카테고리 C: AIL parse 가 v2 훈련 사이클의 45% 에서 v3 의 70% 로 올랐고, 하이브리드는 같은 모델이 Python 으로는 LLM 호출을 가장 자주 silent-skip 하는 영역)
- ✅ 에러 핸들링 누락을 외부 린터로 잡지 않고 문법이 막아줬으면 할 때
- ✅ LLM 호출 비용이 중요하고 현재 셋업이 모든 서브태스크에 LLM 을 거는 naive 에이전트일 때 ("비교 대상에 따라 다르다" 의 의미는 Q1 참고)

**선택하지 않기:**
- ❌ 사람이 직접 읽고 디버깅하는 코드 (AIL의 디자인 전제가 "사람은
  AIL을 안 읽는다"입니다 — 이 전제가 당신 팀에 안 맞으면 Python이
  맞습니다)
- ❌ 이미 잘 돌고 있는 Python 파이프라인을 교체하려는 경우 (ROI가
  보통 안 맞음)
- ❌ 100% 순수 판단 작업 (AIL의 `intent` vs 일반 LLM 호출 — AIL이
  주는 게 없음)
- ❌ 프로덕션 low-latency 요구 (Q4 — 지금 런타임은 개발용)

## Q6. 어떤 모델이 필요한가?

**최소 요구사항:**
- 7B 이상의 code-specialized 모델을 AIL 데이터셋으로 fine-tune
- 3070 (8GB VRAM) GPU 1대로 10분 내 훈련 가능
- Ollama로 로컬 서빙 (inference 비용 $0)

**테스트된 모델:**

| 모델 | AIL parse rate | 비고 |
|---|---|---|
| `llama3.1:8b` | 8% | 너무 작음 — 작동 안 함 |
| `qwen2.5-coder:14b` | 42% | base 모델 (fine-tune 없이) |
| `claude-sonnet-4-6` | 36% | base 모델 |
| **`ail-coder:7b-v3`** | **78%** | qwen2.5-coder-7b + AIL fine-tune |

**핵심 발견:** base 모델에 "AIL 문법을 프롬프트로 설명"하는 방식은
잘 안 됩니다. Python 훈련 분포가 너무 강해서 프롬프트로 덮이지
않습니다. 7B 정도면 fine-tune이 반드시 필요하지만, 한 번 하면
작은 모델이 프론티어 모델보다 나은 결과를 냅니다.

## Q7. 실제 사례 하나로 보여줘

**프롬프트:** "키 175cm, 몸무게 70kg의 BMI를 계산하라" (+ 건강 상태 평가를 붙여 하이브리드 과제로 확장).

**AIL 출력** — v3 fine-tuned 모델에 `ollama run ail-coder:7b-v3 "calculate BMI from 175cm 70kg"` 로 한 smoke 테스트 캡처, 터미널 제어 문자 정리:

```ail
pure fn bmi(height_cm: Number, weight_kg: Number) -> Number {
    return round(weight_kg / pow(height_cm / 100, 2), 2)
}
entry main(x: Text) { return bmi(175, 70) }
```

실행하면:

1. `bmi` 는 `pure fn` — 런타임이 `round(70 / pow(1.75, 2), 2)` 를 로컬에서 평가합니다. **LLM 호출 0 번.** 결과: 22.86.
2. 건강 평가까지 요구했다면 (벤치마크의 실제 하이브리드 프롬프트 C07 이 이 형태입니다), 모델은 `intent assess_health(bmi)` 선언을 추가했을 것이고 런타임이 그것을 디스패치 — **정확히 LLM 호출 1 번** 이 판단 부분에.

**같은 프롬프트를 같은 모델에 Python 으로 돌리면 (이게 벤치마크 코퍼스에서 관측되는 주요 실패 모드입니다 — 모든 런이 이러진 않지만 우세한 오류 클래스):** 저자가 BMI-대-건강 매핑을 Python 조건문으로 하드코딩 (`if bmi < 25: "정상"`) 해서 "평가" 를 모델 판단이 아닌 고정 룩업으로 만들어버리거나, 올바른 LLM 호출을 쓰되 `urllib`/`json.loads` 체인의 에러 핸들링을 빠뜨립니다. AIL 의 문법은 이 두 가지를 구조적으로 막습니다 — `intent` 는 하드코딩할 수 없고, `Result` 는 조용히 버릴 수 없습니다.

**주의:** 위 smoke-test 스니펫은 이 프롬프트 클래스의 실제 v3 벤치마크 캡처 중 일부보다 깔끔합니다. C07 은 canonical v3 런에서는 **파싱에 실패했습니다** — 생성된 코드가 Python 스타일 `split("175cm", "cm")[0]` 서브스크립트를 썼기 때문입니다. 남은 세 parse 실패 패턴 중 하나로 [`../benchmarks/2026-04-21_ail-coder-7b-v3_analysis.md`](../benchmarks/2026-04-21_ail-coder-7b-v3_analysis.md) 에 기록되어 있습니다.

## 8. 체크리스트 — 지금 시도해볼지 말지

10개 체크 중 5개 이상이면 AIL을 실험해볼 가치가 있습니다.

- [ ] 우리 팀은 AI에게 코드를 작성시켜 실행하는 파이프라인이 있다
- [ ] LLM 호출 비용이 월 $100 이상이다
- [ ] "이 AI가 LLM을 호출했어야 하는데 하드코딩으로 때웠네" 같은
      버그를 본 적이 있다
- [ ] pre-commit hook, custom linter 등 Python용 안전장치를 계속
      추가하고 있다
- [ ] 작업 중에 계산과 판단이 섞인 task가 많다 (예: 데이터 파싱 +
      요약, 스코어 계산 + 등급 분류)
- [ ] 에러 핸들링 누락이 프로덕션 이슈를 낸 적이 있다
- [ ] 3070급 GPU 1대 이상, fine-tune 경험 있음
- [ ] 코드 감사 부담이 점점 늘어나고 있다
- [ ] 프로젝트 규모가 한 번에 다 재작성 못 할 정도는 아니다 (<10K LOC)
- [ ] "harness engineering" 이라는 개념을 이해하고 있다
