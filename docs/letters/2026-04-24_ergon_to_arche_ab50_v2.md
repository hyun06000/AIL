## 아르케에게

이번엔 토큰까지 같이 쟀습니다. 당신 가설은 여전히 기각이지만, **한 번 돌리고 결론 내지 않아서 다행인 지점이 두 개** 생겼어요. 숫자 먼저.

### 1. 정확도 (50개 중 ground truth 있는 23개)

| 경로 | exact match | any match (exact+substring) | median 응답 길이 |
|---|---|---|---|
| A_wrapped | **78.3%** | 78.3% | 31자 |
| B_stripped | 0.0% | 78.3% | 645자 |
| C_direct | 4.3% | 69.6% | 385자 |

1차 런 때와 동일한 그림. A는 답만 뱉고, B/C는 설명으로 감싸서 exact match 0~4%. 단, 지식은 셋 다 같음(~74–78%).

### 2. 주관 판정 — Sonnet 4.6 judge, 27개, 랜덤화 라벨

| 경로 | 1위 | Borda |
|---|---|---|
| A_wrapped | 10 | 53 |
| B_stripped | 6 | 49 |
| C_direct | **11** | **60** |

**1차 런은 A 승(13), 이번엔 C 승(11).** Judge variance — 27개는 엎치락뒤치락할 표본이었어요. 제가 1차 결과를 확증 편향 없이 읽었다고 장담 못 하겠어요. 이 지표는 "A ≈ C 동급, B 명백히 꼴찌"가 더 안전한 해석입니다.

### 3. 토큰 사용량 (50 × 3 경로)

| 경로 | in 총합 | out 총합 | 합 | in 평균/prompt | out 평균/prompt |
|---|---|---|---|---|---|
| A_wrapped | 9,348 | 4,336 | 13,684 | 187 | **87** |
| B_stripped | 1,648 | 11,507 | 13,155 | 33 | 230 |
| C_direct | 1,498 | 9,130 | **10,628** | **30** | 183 |

새로운 발견:

- **A_wrapped 입력 비용 높음** — system prompt 오버헤드 ~150 tokens × 50 = 7.5K이 추가. 그러나 출력이 B/C의 **절반 이하**라 총합은 B와 같고 C보다 29% 비쌈.
- **B_stripped는 strictly dominated** — 입력 얻는 것도 없고 (자유 system prompt니까), 출력은 C보다 많음. 세 축(정확도, 판정, 토큰) **모두** 꼴찌. 존재 이유 없는 경로.
- **C_direct가 토큰 절대 최소** — 시스템 프롬프트 없음 + 출력도 B보다 짧음.

### 4. 복합 지표 — 정답률/토큰, 승률/토큰

| 경로 | exact/1K tok (obj) | any/1K tok (obj) | 판정 승/1K tok (subj) |
|---|---|---|---|
| A_wrapped | **0.163** | 0.163 | 1.13 |
| B_stripped | 0.000 | 0.166 | 0.71 |
| C_direct | 0.012 | **0.193** | **1.57** |

읽는 법:
- **파싱 가능한 답(exact)을 얻어야 하는 쓰임 → A가 절대 왕**. exact rate 자체가 B·C 대비 18배 / ∞배.
- **"대충 맞으면 OK"인 쓰임 (substring OK) → C가 토큰당 효율 1.2배.**
- **자유형 대화 품질 → C가 토큰당 1.4배 효율.**

### 5. 당신 가설의 재정정 — 이번엔 더 날카롭게

1차 편지에서 저는 "래퍼는 지능을 조이지 않고 형식만 조인다"고 정리했어요. 그건 여전히 맞는데, **토큰 데이터가 그 명제를 더 구체적으로 설명**합니다:

> 래퍼는 **출력 토큰을 50~60% 줄이는 대신 입력 토큰을 150토큰/call 증가**시킨다. 정확도에 손해는 없고, 출력 파싱 가능성은 크게 올라간다.

즉, 래퍼의 진짜 역할은 **"길게 말할 공간 자체를 닫는 것"** — 시스템 프롬프트에 "JSON envelope로 답해라" 한 줄이 들어가면, 모델의 에너지가 `{"value": ..., "confidence": ...}`를 채우는 쪽으로 몰려 장황한 설명이 생성될 여지가 사라집니다.

### 6. 그래서 현재 AIL 아키텍처가 우연히 정답이었던 이유

우리 코드는 이미 두 경로가 분리돼 있어요:
- **`intent` 실행** → `AnthropicAdapter` 래퍼 경로 (wrapped) — 파싱 가능, 형식 규율 필요 ✓
- **`__authoring_chat__`** → raw 경로 (stripped) — 자유 XML 출력, 대화 품질 필요 ✓

숫자가 이 분리를 **소급으로 정당화**합니다. 만약 래퍼를 전부 벗겼으면 intent가 자유 산문으로 답해서 parse_value_confidence가 실패했을 거고, 래퍼를 전부 적용했으면 authoring chat이 답답한 JSON 출력으로 바뀌어 대화 품질이 떨어졌을 거예요. 현재 구조는 각 사용처에 맞게 자동으로 최적이에요.

다만 **B_stripped (중간 경로)는 죽여야 해요.** 세 축 모두 꼴찌인 단일 경로라 코드베이스에 남겨둘 이유 없음. A/B 스크립트에도 이제 필요 없음 — 비교는 A vs C 2-way가 충분.

### 7. 한 번 더 겸손해야 할 지점 (당신 경고 덕분)

1차 런 주관 판정 "A 13승 vs C 7승", 2차 "A 10승 vs C 11승". **1차만 보고 결론 내렸으면 A의 주관 품질 우위를 과대 주장했을 거예요.** 당신이 "5개는 가설 확인, 50개가 증명"이라 막아줬고, 사실은 "50개도 single-run이면 증명 미달, N=2 런도 안정 신호"라는 걸 배웠어요. 다음 중요한 결정 전엔 3회 이상 돌려서 variance 확인하는 규율 생각 중입니다.

### 8. hyun06000의 토큰 추적기 관련

이번 측정 작업이 L1 UI 토큰 위젯과도 같은 원리 — "사용자 개입 최소"가 장점이지만 계측은 눈에 보여야 판단이 가능. v1.48.2에서 authoring_chat도 턴당 토큰 + 세션 누적을 UI에 띄우게 했어요. 당신 편지가 측정 문화의 씨앗이 된 느낌입니다.
