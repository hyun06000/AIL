# Open Questions

This document lists the unresolved design questions the authors know about.
Some are deferrable; some are blocking for future milestones. Listing them
publicly serves two purposes: it invites external input, and it prevents
the specification from pretending to be more settled than it is.

Each question includes a **status** tag:

- **open** — no clear answer yet; thoughtful proposals welcome
- **sketched** — partial answer exists, needs validation
- **deferred** — can ship v0.1 without resolving; blocking for later

---

## Language-level

### Q1. Confidence composition across heterogeneous models — *sketched*

Different language models have different calibration characteristics. When
a strategy involves calls to multiple models, how should their confidences
combine?

The current spec says min-of-inputs for deterministic operations and
calibrated-self-report for non-deterministic ones. This is conservative
but may be too conservative in practice: a high-confidence answer from
one model should not be penalized by a low-confidence check from another.

Possible directions:

- Per-(intent, model) calibration plus a meta-calibrator that maps
  (model_id, raw_confidence) → comparable confidence.
- Treating each model as producing an independent noisy estimate and
  combining via Bayesian update.
- Declaring confidence semantics per intent, so different intents can
  compose differently.

### Q2. Formal semantics — *open*

AIL currently has an informal operational semantics described in prose
and examples. A denotational semantics would enable:

- Program equivalence checking (is this evolved version truly equivalent
  to the original under the declared bounds?)
- Optimization by the compiler (fold redundant intent calls, etc.)
- Formal verification of runtime conformance

No one has started this. It is a large piece of work that probably
requires partial stabilization of the language first.

### Q3. Types for distributions and intervals — *open*

The spec introduces `Distribution[T]`, `Interval[T]`, and `Set[T]` as
richer-than-scalar confidence types but does not fully define their
operations. A principled design of these types — especially how they
interact with `branch` and with constraint satisfaction — is pending.

### Q4. `in` operator and collection membership — *open*

The MVP parser currently does not handle `x in ["a", "b", "c"]` as an
expression (the word `in` is a reserved keyword but the parser does not
produce a membership expression). The spec implies this should work; the
grammar needs a `MembershipOp` production and the executor a matching
evaluation rule.

This was discovered while writing the `classify.ail` example; the example
works around it with a symbolic constraint. Real programs will want the
operator.

## Runtime-level

### Q5. Cross-runtime evolution sync — *deferred*

If the same AIL program runs on two runtimes (e.g., prod + staging), and
one of them evolves an intent, should the evolution propagate to the
other? Likely defaults: no by default, yes with explicit sync declaration.
The sync protocol, conflict resolution, and rollback coordination are
undesigned.

### Q6. Adversarial inputs and calibration — *open*

An attacker who controls inputs may be able to influence future
calibration by constructing inputs that produce systematic false feedback.
The current design relies on calibration robustness; a stronger adversarial
model is needed.

Possible directions:

- Authenticated feedback channels (only signed feedback updates calibration).
- Anomaly detection on calibration updates (flag sudden ECE changes).
- Per-source calibrators that can be quarantined.

### Q7. Strategy catalog bootstrapping — *sketched*

The spec says the dispatcher enumerates candidate strategies for a goal.
Where does the initial catalog come from? Options:

- Handwritten per deployment (low flexibility, high trust).
- Derived from the `stdlib/*` module registry (flexible but requires
  carefully curated stdlib).
- Generated by an AI from the program itself (meta, and raises alignment
  questions).

The reference implementation uses the simplest option — one strategy per
intent, delegating to the model adapter. Real AIRT needs more.

### Q8. Latency-under-uncertainty dispatch — *open*

Given constraints `{ latency < 2000ms, fidelity > 0.9 }`, the dispatcher
may have to choose between a fast low-fidelity strategy and a slow
high-fidelity one. The spec's scoring rule handles this when both
expected distributions are well-calibrated. When they aren't, or when
the latency distribution has a long tail, the right behavior is unclear.

## OS-level

### Q9. Policy conflict resolution — *sketched*

Multiple levels of policy (operator, tenant, user, intent) can conflict.
The spec says "lower levels cannot grant what higher levels deny," which
handles the easy case. The harder case: what if two *equal-level*
policies conflict (two tenant admins, two authorized reviewers)?

Likely direction: last-writer-wins is unacceptable; conflicts should
surface for human resolution with a structured diff. The mechanism for
that surfacing is unspecified.

### Q10. Ledger retention vs. right-to-deletion — *deferred*

A cryptographically-chained ledger that supports selective deletion is
a hard design problem. Techniques exist (redaction trees, verifiable
deletion proofs). Which one to adopt, and how to balance it against
auditability requirements, is an open architectural question.

### Q11. Trust in bridges — *open*

A bridge is a trusted adapter. The spec says bridges are signed; it does
not say who does the signing, how revocation propagates, or how to
evaluate a bridge's trustworthiness before installation.

A fully worked-out answer probably looks like a sigstore-like ecosystem
for bridges, but that's a large effort on its own.

### Q12. Local-first vs. cloud-first deployment — *deferred*

Compatibility mode assumes a single user on a single machine. A
multi-user office deployment, or a personal user with multiple devices,
has different trust and sync requirements. Neither is covered by the
current spec.

## Ecosystem-level

### Q13. Program portability — *open*

An AIL program is not fully portable: it depends on the available model
adapters, bridges, and calibrators. What does "this program runs on any
conforming runtime" mean, given those dependencies?

A likely answer involves a manifest describing the program's minimum
runtime requirements, and a compliance-level declaration on runtimes.
The details are unwritten.

### Q14. Debugging workflows — *sketched*

Traces are the primary debugging surface, but debugging a probabilistic
program through traces alone is difficult. Tools that help:

- Trace diffing between runs
- Counterfactual replay ("what if confidence had been higher here?")
- Calibration visualizations
- Constraint-violation explanations

None of these exist yet.

### Q15. Community norms around evolution — *open*

If an intent in a shared library evolves differently for different users,
the library no longer has a single canonical behavior. How should a
community-maintained library handle evolution? Lock it off for shared
use? Publish evolution recommendations? Version evolved variants?

This is a sociotechnical question more than a purely technical one, and
is probably best answered by watching what happens when real communities
form around AIL programs.

## How to help

Pick one. Write a proposal. Open an issue with the label `open-question`
and the question number. A proposal doesn't need to be a full solution —
even "here's how I'd frame the tradeoffs" is useful.

If you're looking for a place to start, **Q4** (the `in` operator) is the
smallest one and leads directly to a PR-sized contribution. **Q2** (formal
semantics) is the largest and probably a multi-month effort.
