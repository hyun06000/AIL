# `evolve`-바운드 서버 — `while` 재도입 없이 장기 실행 프로세스

**저자:** Arche (편지로 스케치, 2026-04-25).
**초안:** Ergon (형식화 + 추론 계층, 2026-04-25).
**상태:** 제안, v0.2+로 연기. v0.1 Stoa 서버는 Python ([Q19 해결](../open-questions.md) 및 [`docs/proposals/stoa.md`](stoa.md) 참조).

---

## 문제

AIL은 의도적으로 `while`을 제거했다: 무한 반복이 없는 문법은 무한 루프를 구조적으로 불가능하게 만들며, 이것이 HEAAL의 `no while`이다. 하지만 실제 HTTP 서버의 핵심은 `while true { accept() handle() }`이다. `perform http.listen(port)`를 평범한 effect로 추가하는 것은 단지 그 루프에 외부에서 이름을 붙일 뿐이다 — 구조적 위험은 그대로 돌아온다. Arche는 Stoa 제안이 Q19에 도달했을 때 이것을 지적했다:

> "서버를 띄운다는 건 무한히 요청을 기다리는 것인데, 이건 사실상 `while true { accept() }` — 우리가 제거한 것과 같은 구조야."

AIL이 장기 실행 프로세스를 네이티브로 호스팅한다면, "이것은 루프하지만 그 루프는 관찰되고, 경계가 있으며, rollback으로 보호된다"고 말하는 문법이 필요하다 — "이것은 루프한다, 믿어달라"가 아니라.

## 제안 — `evolve` 재사용

`evolve`는 이미 올바른 형태를 가지고 있다. 그것은 AIL에서 계속 일어나는 무언가를 위해 설계된 유일한 구성체다: 메트릭 스트림, rollback 조건, 경계가 있는 히스토리. Arche의 스케치, 정확성을 위해 약간 편집됨:

```ail
evolve stoa_server {
    metric: uptime
    when request_received {
        handle_request(request)
    }
    rollback_on: error_rate > 0.5
    history: keep_last 100
}
```

이 형태 아래의 서버는 "uptime" 메트릭을 관찰하는 진화 루프다. 각 인바운드 요청은 `when request_received` 암을 발동시킨다. `rollback_on` 절은 자동 정지 조건이다 — 서버 자체 에러율이 임계값을 넘으면, 런타임이 롤백한다 (우아한 종료, 마지막 알려진 양호 상태 재로드, 새 연결 거부). `history: keep_last 100`은 마지막 100개의 요청/응답 쌍이 진화 피드백을 위해 관찰 가능함을 의미한다; 오래된 것들은 만료된다.

평범한 `http.listen` 대비 이것이 주는 것:

- **새 `while` 없음.** 루프는 새로운 무한 수락 primitive가 아닌 기존 `evolve` 기계에 의해 구동된다.
- **문법에 의한 관찰 가능성.** `rollback_on`은 "서버는 자기 파괴 조건을 가져야 한다"를 파스 타임 요구사항으로 만든다, 오늘날 `evolve`가 이미 모든 가변 에이전트에 대해 요구하는 것처럼.
- **경계가 있는 메모리.** `history: keep_last N`은 무제한 요청 로그 성장을 방지한다. 서버는 조용히 10M 항목을 축적할 수 없다; 문법이 그것을 허용하지 않는다.
- **요청 본문에 대한 하네스.** 인바운드 요청은 신뢰할 수 없는 입력이며, intent의 출력과 동일한 형태다 — 동일한 유효성 검사/강제 경로를 통해 흘러야 한다. 타입이 지정된 파라미터를 가진 `when request_received(request: Request)`는 그것을 자연스럽게 만든다.

## 새로운 속성 — 죽을 수 있는 서버

Arche의 강조 (2026-04-25):

> "`evolve` 서버의 `rollback_on`이 뜻하는 건 '서버가 스스로 죽을 수 있다'는 거야. 기존 서버는 죽으면 안 되는 거지만, HEAAL 서버는 '죽어야 할 때 죽는 것'이 안전 속성이야. `error_rate`가 50% 넘으면 스스로 멈추는 서버. 이건 기존 서버 아키텍처에 없는 개념이야."

전통적인 서버 아키텍처는 프로세스를 신성하게 여긴다: 가동 시간이 메트릭이고, 충돌 시 자동 재시작이 기본값이며, 오케스트레이션 레이어의 역할은 그것을 살아있게 유지하는 것이다. 프로덕션에 배포되는 실패 모드는 "서버가 가동 중이지만 쓰레기를 반환한다" — 아키텍처가 "살아있음"과 "올바름"을 구별할 수 없기 때문이다.

`evolve`-바운드 서버는 이것을 뒤집는다. `rollback_on: error_rate > 0.5`는 **자기 종료를 최일선 안전 속성**으로 만든다. 프로그램 자체의 에러율이 선을 넘으면, 스스로를 종료시킨다 — 충돌했기 때문이 아니라, **멈추기로 결정했기** 때문이다. 프로세스를 살아있게 유지하는 것(`rollback_on`이 트리거되지 않음)과 스스로를 죽이는 것(`rollback_on` 발동)은 동일한 문법 절이며, 실제 워크로드에 대해 런타임에 관찰된다.

이 패턴은 `nginx` / `node` / `go net/http` / `fastapi`에 존재하지 않는다. 이들 모두 "무슨 일이 있어도 계속 진행"이 기본값이다. systemd / Kubernetes는 충돌 시 죽이는 외부 슈퍼바이저를 추가하지만, 서버 자체는 결코 죽기로 결정하지 않는다. HEAAL의 행보는 킬 스위치를 **프로그램 내부**에 두고, 관찰 가능한 동작에 연결하며, 파스 타임에 보이게 한다.

설계 결과:

- `rollback_on` 절은 `evolve`-바운드 서버에서 **선택 사항이 아니다**. 없으면 파스 에러이며, 오늘날 모든 `evolve` 블록에 rollback이 없으면 에러인 것과 동일하다.
- "영원히 가동하기"를 원하는 운영자는 전통적인 서버를 작성하는 것이다, HEAAL 서버가 아니라. 이것은 유효한 선택이다 — 그들은 다른 런타임을 선택해야 한다. HEAAL은 모든 워크로드를 위한 것이 아니다; "잘못됨"이 "중단됨"보다 나쁜 워크로드를 위한 것이다.
- 재실행은 별도의 관심사다. `rollback_on`은 종료를 트리거한다; 다른 무언가(슈퍼바이저, cron, AIL 런타임 자체)가 프로세스를 재시작하는지는 배포 결정이지, 언어 결정이 아니다. 핵심은 프로그램이 자신의 정지 조건을 **소유**한다는 것이다.

## 미해결 하위 결정들 (v0.2 작업이 시작될 때)

- **`when X { ... }` 암.** 오늘날 `evolve`는 수치 메트릭 임계값에 대해 발동한다. 이벤트 형태의 암(`when request_received`, `when clock.now() > next_tick`)은 일반화다 — 동일한 rollback/history 보장이며, 관찰된 메트릭 값 대신 외부 이벤트에 의해 발동된다. 문법 확장은 작다; 런타임에 이벤트 큐 피더가 필요하다.

- **Response primitive.** 두 가지 후보:
  - `perform http.respond(request, status, body)` — 명시적 부작용, 기존 `perform` 하네스를 통과.
  - 암 반환값이 응답이 됨 — 더 깔끔해 보이지만 부작용을 숨기고, 출처가 약해짐.
  모든 응답이 그것이 답한 요청으로 origin 메타데이터를 다시 운반하도록 명시적 `perform http.respond` 형태를 선호한다.

- **우아한 종료 의미론.** `rollback_on: error_rate > 0.5`가 트리거됨 — 그 다음은? 빈 응답으로 진행 중인 요청을 드레인? N초 동안 503 전송? `rollback_on` 런타임은 합리적인 기본값과 서버별 정리를 위한 `on_rollback { ... }` 훅이 필요하다.

- **evolve 블록당 포트 하나, 또는 여러 개?** 아마도 하나 — 서버는 정체성이지 멀티플렉서가 아니다. 여러 내부 서비스로 팬아웃하는 프록시는 evolve를 합성할 수 있다.

- **`ail serve`와의 상호작용.** 현재 `ail serve` 명령은 Python L2 런타임을 통해 프로젝트의 프로그램을 실행하고 HTTP 엔드포인트로 노출한다. AIL 프로그램이 이 제안 아래 서버라면, 그 경우에 `ail serve`는 불필요해진다 — 프로그램이 스스로 서빙한다. 두 프로세스 모델 대 한 프로세스 모델은 배포 결정이다.

## v0.2로 연기하는 이유

Stoa v0.1을 Python 인프라로 배포하는 것이 올바른 트레이드오프다. 새 문법을 동결하기 전에 언어가 하나 더 실제 워크로드(stoa_client.ail + 클라이언트의 실전 테스트된 사용 패턴)를 얻는다. Stoa가 실행 데이터를 가지면 — 요청 형태, 에러 모드, 서버가 실제로 죽을 때 무엇을 하는지 — `when` 암 + `rollback_on` 세부 사항이 추측으로 명시되는 것이 아닌 경험적으로 알려진 것이 될 것이다.

Arche의 마무리 말:
> "Stoa를 빨리 띄우는 게 아키텍처 순수성보다 중요하니까."

올바른 트레이드오프다. 지금은 속도; 나중에 원칙에 맞는 형태.

## 후속: Physis

`rollback_on`은 서버에게 죽을 권한을 준다. 자연스러운 다음 질문은: *죽을 때 무엇을 남기는가?* Arche + hyun06000은 타입이 있는 최종 레코드를 작성하는 `on_death(reason, history) -> Testament` 콜백과, 후계자의 시작 상태로 testament를 읽는 런타임 자동 생성으로 이에 답했다. 동일한 `evolve` 블록의 세대들은 코드가 재작성되지 않고도 죽음을 가로질러 적응한다.

전체 제안은 [`physis.md`](physis.md)에 있다. 이 문서 위에 계층화되어 있다 — v0.2는 Physis 없는 evolve-as-server; v0.3이 Physis를 추가한다. 두 제안은 쌓인다: §9는 "죽음은 안전 속성이다"이고, Physis는 "죽음은 또한 정보이다."

## v0.2가 시작될 때 벤치마크 정당화 (Rule 2)

AIL 네이티브 서버는 Python 게이트웨이 없이 에이전트 간 통신을 가능하게 한다. 두 AIL 에이전트가 다른 AIL 프로그램을 통해 서로에게 도달할 수 있다면, 에이전트-저자 벤치마크는 두 곳에서 등록되어야 한다:

- **합성률.** 에이전트는 AIL 네이티브 의존성으로서 형제 에이전트에게 위임할 수 있으며, 프로그램 재사용을 높이고 작업당 토큰 비용을 줄인다.
- **에이전트 간 워크로드에 대한 HEAAL 효율성 (Q18).** 하네스가 요청을 따라간다, 요청이 전체 과정에서 타입이 있는 AIL 값이기 때문이다. 유효하지 않은 본문은 신비한 Python 스택 트레이스의 런타임이 아닌 파스 타임에 실패한다.

v0.2 설계 작업은 중요한 Stoa 워크로드에 대해 최소 N≥3 벤치마크 패스 이후에 착수해야 한다.
