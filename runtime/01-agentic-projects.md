# Agentic Projects — INTENT.md as the First-Class Surface

**Version:** 0.1 vision document · **Status:** Pre-implementation. Spec only. Recorded so the next session can pick up the thread after Stage D closes.

This document captures a design conversation between hyun06000 and Claude Code on 2026-04-22 about what an AIL "project" should look like once the language is no longer just a one-shot CLI calculator. It belongs to the L2 layer (AIRT runtime — see [`00-airt.md`](00-airt.md)) and answers a specific question: *if AIL is supposed to let non-developers ship software, what does the artifact on disk look like?*

The short answer: an AIL project is a folder with one human-edited markdown file (`INTENT.md`) and an in-project AI agent that owns everything else.

---

## 1. The premise — non-developers don't use terminals

The original AIL vision was "humans never touch the code layer." A natural-language prompt goes in, a result comes out. We currently honor this through `ail ask "..."` for one-shot answers.

But:

- One-shot calculators are not products. People want services that keep running.
- Long CLI flag chains (`ail ask "X" --serve --port 8080 --test "..." --test "..."`) are a non-developer hostile surface. Typing in a terminal is the friction itself.
- A `.ail` source file is still a *programmer's artifact* — it has braces, type signatures, and a grammar. Non-developers should not be expected to read it.

The deepest version of "humans never touch the code layer" is: **humans don't even own the code file.** The AI does. The human owns a natural-language description of what they want, and the AI maintains the program from that description.

This is exactly the Claude Code interaction pattern: a human writes a `CLAUDE.md`, asks for things in plain language, and an agent in the project handles file creation, editing, execution, and debugging within the permission boundary.

We adopt this pattern as the project structure for AIL itself.

---

## 2. Project layout

```
my-summarizer/
├── INTENT.md              # the only file the human writes
├── app.ail                # AI-owned: generated and maintained from INTENT.md
└── .ail/
    ├── tests.json         # extracted from INTENT.md by the agent
    ├── ledger.jsonl       # append-only record of every agent action and evolve event
    └── state/             # cross-session evolve state (finally persistent)
```

**Promises:**

- The human edits `INTENT.md`. Optionally nothing else, ever.
- `app.ail` is autogenerated and re-generated. The human may read it (`ail show source`), but is not expected to.
- `.ail/state/` solves the long-standing evolve persistence gap — the runtime now has a place to put history that survives process restart.
- `.ail/ledger.jsonl` is the trace ledger lifted out of memory and into durable storage. Every authoring decision, every test pass/fail, every evolve trigger, every rollback is appended here. This is the MLOps audit trail by default.

---

## 3. INTENT.md format

Plain Markdown. No new syntax. We define a conventional set of section headers; the agent extracts structure from those.

```markdown
# Summarizer

받은 텍스트를 한 문장으로 요약하는 서비스.

## 동작
- 영어/한국어 둘 다 지원
- 빈 입력 → 에러
- 10자 미만 → 에러

## 테스트
- "Long article about cats..." → 한 문장 요약
- "" → 에러
- "hi" → 에러

## 배포
- 포트 8080
```

**Conventional sections (English / Korean variants both honored):**

| Section | Purpose | Example |
|---|---|---|
| (preamble) | One-paragraph what-is-this | "받은 텍스트를 한 문장으로 요약하는 서비스" |
| `## 동작` / `## Behavior` | Bullet list of behavioral requirements | "빈 입력 → 에러" |
| `## 테스트` / `## Tests` | Sample inputs with expected outcome shape | `"hi" → 에러` |
| `## 배포` / `## Deployment` | Port, environment, schedule | `포트 8080` |
| `## 진화` / `## Evolution` | (optional) when to retune, rollback conditions | `정확도 < 0.8 N회 연속 → 재튜닝` |

**Why Markdown not YAML/TOML/custom:**
Non-developers have seen Markdown (every blog platform, every README, every chat tool that supports formatting). YAML's significant whitespace bites them. TOML's `[section]` headers feel programmer-ish. Markdown headers + bullet lists are the lowest-friction structured format that exists.

**Why conventional sections not a strict schema:**
The agent is reading natural language anyway. A strict schema would force the human to think like a programmer ("must have exactly these keys"). Convention with fallback (the agent makes a reasonable interpretation if a section is missing) is more forgiving.

## 4. Commands — only two, by design

```bash
ail init <name>     # create folder + INTENT.md template
ail up              # read INTENT.md → agent generates/updates app.ail →
                    # extracts tests → runs tests → if all pass, serves on configured port
                    # → watches files → auto-reloads on edit
```

That is intentionally the entire surface for the non-developer path. Two commands, one editable file. Live-reload is not a flag — it is the default, because the audience does not know what live-reload is.

**Future additions (not in v0):**
- `ail chat` — talk to the project in natural language without editing INTENT.md directly. The agent updates INTENT.md *and* app.ail in one move and explains what it did.
- `ail show source` — print app.ail (the curious developer's escape hatch).
- `ail bundle` — single-file executable for true OS-level double-click. L3 territory.

**What `ail up` actually does (state machine):**

```
   ┌──────────────────────────────────────────────────────────┐
   │  read INTENT.md                                            │
   └──────────────────────────┬───────────────────────────────┘
                              ▼
   ┌──────────────────────────────────────────────────────────┐
   │  extract: behavior bullets, test cases, deployment config │
   │  → write/update .ail/tests.json                            │
   └──────────────────────────┬───────────────────────────────┘
                              ▼
   ┌──────────────────────────────────────────────────────────┐
   │  if app.ail empty or INTENT.md changed since last gen:    │
   │    author model writes/updates app.ail                    │
   │    parse + purity check (existing AIL pipeline)           │
   │    retry on parse error (existing retry loop)             │
   └──────────────────────────┬───────────────────────────────┘
                              ▼
   ┌──────────────────────────────────────────────────────────┐
   │  run all extracted tests                                  │
   │  on failure: agent diagnoses, edits app.ail, retries      │
   │  every action appended to .ail/ledger.jsonl               │
   └──────────────────────────┬───────────────────────────────┘
                              ▼
   ┌──────────────────────────────────────────────────────────┐
   │  start HTTP server on configured port                     │
   │  POST /  →  entry main(input) of app.ail                   │
   │  watch INTENT.md and app.ail; reload on change            │
   └──────────────────────────────────────────────────────────┘
```

## 5. Why this closes multiple open problems at once

This design lands in the middle of four otherwise-separate gaps:

1. **Non-developer deliverable.** A folder with INTENT.md is the artifact. `ail up` is the runtime. The non-developer never types AIL.
2. **`evolve` persistence.** `.ail/state/` is the cross-session evolve store. The MLOps demo we want to build (drift → retune → rollback) finally has a substrate.
3. **MLOps differentiator.** `.ail/ledger.jsonl` is the lifecycle audit log built into the project structure. No external MLflow/W&B required — the language ships with it.
4. **L2 first concrete step.** `runtime/00-airt.md` describes intent-graph execution as a long-running activity. Up to now we have only run intent graphs as one-shot CLI invocations. `ail up` is the first time AIL programs become services.

Four gaps, one design.

## 6. Implementation stages

**v0 — minimal closure (target: 1 week)**
- `ail init <name>` writes folder + INTENT.md template.
- `ail up` does the state machine above without live-reload (no file watching). The agent generates app.ail once, runs tests, serves until Ctrl-C.
- Author model is the existing `ail ask` author (`AIL_AUTHORING_BACKEND` etc.).
- Test extraction is naive: parse `## 테스트` / `## Tests` bullets as `(input, expected_outcome)` pairs.
- Server: stdlib `http.server` is fine; one route `POST /` calls `entry main(input)`.
- `.ail/ledger.jsonl` records each authoring attempt + each test run + each request served.

**v1 — interactive closure (target: 2-3 weeks)**
- File watching + automatic reload on INTENT.md or app.ail change.
- `ail chat` — natural-language editing without touching INTENT.md directly. The agent diffs INTENT.md, regenerates app.ail, re-runs tests, explains what it did.
- evolve state actually persisted to `.ail/state/`.

**v2 — autonomous closure (target: open)**
- Agent diagnoses test failures and produces minimal app.ail patches without re-authoring from scratch.
- Agent surfaces drift signals from `.ail/ledger.jsonl` ("the model's confidence on these inputs has dropped 30% over the last 1000 requests") and proposes evolution.
- Optional: rich web UI on a separate port for ledger / evolution / test browsing.

## 7. Open decisions to settle before v0 starts

1. **In-project agent: self-built or Claude Agent SDK?**
   - Self-built keeps zero anthropic-only dependency, preserves HEAAL's model-independence promise, takes longer.
   - Claude Agent SDK reaches v1 faster but binds the agent layer to Anthropic. Other model families (the GPT-4o / Gemini transferability we've started measuring) would need parallel implementations.
   - **Recommendation:** v0 and v1 self-built (the agent loop at v0/v1 is small enough). Reserve the SDK option for v2's autonomous diagnosis where the gap matters.

2. **Single file vs folder for non-trivial projects.**
   - v0 spec assumes one app.ail per project. Multi-file projects (stdlib reuse, sub-intents in their own files) need a story.
   - **Recommendation:** punt to v1. Single file is enough for the demos that prove the model.

3. **Port collision and process supervision.**
   - `ail up` claims a port. Two `ail up` from different folders on the same port collide.
   - **Recommendation:** v0 fails loudly with a clear message. No supervision, no port allocation. v1 adds `ail status` / `ail down`.

4. **Where does the author model run for non-developers without API keys?**
   - The Quick Start path uses Anthropic. Non-developers may not have any API key.
   - The local fallback is `ail-coder:7b-v3` via Ollama, but that requires Ollama installed.
   - **Recommendation:** v0 documents both paths; v1 considers a hosted "intent endpoint" the AIL project could provide as a default. (This is a project-level decision for hyun06000, not a code change.)

5. **INTENT.md schema: do we publish a strict spec or stay informal?**
   - Strict schema enables tooling (validators, IDE plugins). Informal schema preserves the "just write what you want" feel.
   - **Recommendation:** ship informal in v0; if external users start sharing INTENT.md projects and convergent patterns emerge, formalize then.

---

## 8. Connection to L1 and L3

**L1 (AIL language):** No new grammar. INTENT.md is a project-level construct above the language. The author model still emits standard AIL.

**L3 (HEAAOS):** The `.ail/` directory is the seed of what becomes a HEAAOS *intent-as-process* primitive — an OS where a "running program" is an INTENT.md + agent + ledger triple, not an ELF binary + threads. This document does not implement L3, but the project layout is designed to be liftable.
